Chapter 01

WHY OOP IN TYPESCRIPT?

TypeScript brings static typing to JavaScript OOP — meaning the compiler catches class design mistakes before runtime. You get all of JavaScript's flexibility plus Java/C#-style class safety.

"TypeScript is what happens when you give JavaScript a blueprint. OOP in TypeScript is the art of building systems that can grow without falling apart."
🔒

Encapsulation

Bundle data + behaviour together. Control what the outside world can see with private, protected, public. TypeScript adds readonly on top.

🧬

Inheritance

Child classes extend parents. Reuse + specialize. TypeScript checks the hierarchy at compile time so broken override signatures are caught instantly.

🎭

Polymorphism

One interface, many shapes. TypeScript's structural typing means any object that looks like an interface satisfies it — duck typing with guardrails.

🌫️

Abstraction

abstract classes + interfaces let you define contracts without implementations. TypeScript enforces these contracts at compile time.

📝
TypeScript vs JavaScript OOP

JS has OOP via prototypes and ES6 classes. TypeScript adds: access modifiers, abstract, readonly, interface, generics, strict null checks, and full type inference on all class members.

Chapter 02

CLASSES — BLUEPRINTS

A class is a reusable blueprint for creating objects. TypeScript classes compile to ES6 classes but add compile-time type checking on every member.

BankAccount.ts — fully annotated class
// ── Class declaration ──────────────────────────────────────────
class BankAccount {

  // ── Instance fields (each object gets its own copy) ──────────
  private   owner:    string;
  private   balance:  number;
  readonly  accountId: string;   // set once, never changed
  protected currency: string;

  // ── Static field (shared by ALL instances) ───────────────────
  private static instanceCount: number = 0;

  // ── Constructor ───────────────────────────────────────────────
  constructor(owner: string, initialBalance: number = 0, currency: string = "USD") {
    this.owner     = owner;
    this.balance   = initialBalance;
    this.currency  = currency;
    this.accountId = `ACC-${++BankAccount.instanceCount}}`;
  }

  // ── Instance methods ──────────────────────────────────────────
  deposit(amount: number): void {
    if (amount <= 0) throw new Error("Deposit must be positive");
    this.balance += amount;
  }

  withdraw(amount: number): boolean {
    if (amount > this.balance) return false;
    this.balance -= amount;
    return true;
  }

  // ── Getter / Setter ───────────────────────────────────────────
  get currentBalance(): number        { return this.balance; }
  get ownerName():      string        { return this.owner; }

  set ownerName(name: string) {
    if (!name.trim()) throw new Error("Name required");
    this.owner = name.trim();
  }

  // ── Static method ────────────────────────────────────────────
  static getTotalAccounts(): number { return BankAccount.instanceCount; }

  // ── toString equivalent ───────────────────────────────────────
  toString(): string {
    return `[${this.accountId}] ${this.owner} — ${this.currency} ${this.balance.toFixed(2)}`;
  }
}

// ── Usage ─────────────────────────────────────────────────────
const acc1 = new BankAccount("Alice", 5000);
const acc2 = new BankAccount("Bob",   1200, "EUR");

acc1.deposit(500);
acc1.withdraw(200);

console.log(acc1.toString());        // [ACC-1] Alice — USD 5300.00
console.log(acc1.currentBalance);  // 5300 (via getter)
console.log(BankAccount.getTotalAccounts()); // 2

TypeScript Parameter Properties Shortcut

TypeScript lets you declare and initialize fields directly in the constructor signature. This is unique to TypeScript — it's a massive time saver.

parameter-properties.ts
// ❌ Verbose (Java-style) — don't need this in TypeScript
class Verbose {
  private name: string;
  private age:  number;
  constructor(name: string, age: number) {
    this.name = name;
    this.age  = age;
  }
}

// ✅ Concise — TypeScript parameter properties
class Person {
  constructor(
    private  readonly name: string,
    private           age:  number,
    public            role: string = "user"
  ) {}   // body is empty — fields auto-created!

  greet(): string {
    return `Hi, I'm ${this.name}, age ${this.age} (${this.role})`;
  }
}

const p = new Person("Alice", 30);
console.log(p.greet());  // Hi, I'm Alice, age 30 (user)
// p.name = "Bob";  ← TS ERROR: readonly!
💡
Always Use Parameter Properties

In TypeScript, parameter properties are idiomatic. The access modifier in the constructor parameter automatically declares and initializes the field. Use readonly for fields that shouldn't change after construction.

CONSTRUCTORS & OVERLOADS

TypeScript supports constructor overloads — multiple call signatures for one constructor. Unlike Java, there's only one implementation body.

constructor-overloads.ts
class Vector2D {

  // ── Overload signatures (no body) ───────────────────────────
  constructor(x: number, y: number);
  constructor(magnitude: number, angleDeg: number, polar: true);
  constructor(other: Vector2D);  // copy

  // ── Implementation (handles all overloads) ──────────────────
  constructor(
    public x: number | Vector2D,
    public y: number = 0,
    polar?: true
  ) {
    if (typeof this.x === "object") {  // copy constructor
      const src = this.x as Vector2D;
      this.x = src.x;
      this.y = src.y;
    } else if (polar) {              // polar → cartesian
      const rad = (this.y * Math.PI) / 180;
      this.y = this.x * Math.sin(rad);
      this.x = this.x * Math.cos(rad);
    }
    // else: normal x,y — already set by parameter properties
  }

  add(v: Vector2D): Vector2D {
    return new Vector2D(this.x + v.x, this.y + v.y);
  }

  get magnitude(): number {
    return Math.sqrt(this.x ** 2 + this.y ** 2);
  }
}

const v1 = new Vector2D(3, 4);        // cartesian
const v2 = new Vector2D(v1);         // copy
console.log(v1.magnitude);           // 5

Factory Pattern via Static Methods

factory-method.ts
class Color {
  // Private constructor — can't use 'new Color()' directly
  private constructor(
    public readonly r: number,
    public readonly g: number,
    public readonly b: number
  ) {}

  // Named factory methods — clearer intent than raw constructor
  static fromRGB(r: number, g: number, b: number): Color {
    if ([r,g,b].some(v => v < 0 || v > 255))
      throw new RangeError("RGB values must be 0-255");
    return new Color(r, g, b);
  }

  static fromHex(hex: string): Color {
    const h = hex.replace(/^#/, "");
    return new Color(
      parseInt(h.slice(0,2),16),
      parseInt(h.slice(2,4),16),
      parseInt(h.slice(4,6),16)
    );
  }

  static readonly RED   = Color.fromRGB(255, 0, 0);
  static readonly GREEN = Color.fromRGB(0, 255, 0);
  static readonly BLUE  = Color.fromRGB(0, 0, 255);

  toHex(): string {
    return `#${[this.r,this.g,this.b]
      .map(v => v.toString(16).padStart(2,"0"))
      .join("")}`;
  }
}

const c1 = Color.fromHex("#1a2b3c");
const c2 = Color.RED;
console.log(c2.toHex());  // #ff0000

class Employee {
  public    name:      string;  // accessible everywhere
  protected salary:    number;  // accessible in this class + subclasses
  private   password:  string;  // this class ONLY (TS compile check)
  #ssn:               string;  // TRUE private — runtime enforced (ES2022)
  readonly  id:        string;  // can't reassign after construction
  static    company:   string = "Acme Corp";  // shared

  constructor(name:string, salary:number, ssn:string) {
    this.name     = name;
    this.salary   = salary;
    this.password = hashPassword(ssn);
    this.#ssn     = ssn;
    this.id       = `EMP-${Date.now()}`;
  }

  // Getter gives read access without exposing the field directly
  get annualSalary(): number { return this.salary * 12; }
}

class Manager extends Employee {
  giveRaise(amount:number) {
    this.salary += amount;  // ✓ protected is accessible
    // this.password       ← TS ERROR: private
    // this.#ssn           ← TS ERROR: #private not inherited
  }
}

const emp = new Employee("Alice", 5000, "123-45-6789");
console.log(emp.name);         // ✓ Alice
// emp.salary    ← TS ERROR: protected
// emp.password  ← TS ERROR: private
// emp.id = "x"  ← TS ERROR: readonly
⚠️
private vs #private — Know the Difference

TypeScript's private keyword is a compile-time-only restriction — at runtime, the field is still accessible via (obj as any).field. The ES2022 #field syntax creates a true runtime private field that cannot be accessed externally at all, even with casting. For sensitive data like tokens or SSNs, prefer #field.

inheritance-deep-dive.ts
// ═══ BASE CLASS ══════════════════════════════════════════════
abstract class BaseProduct {
  private static _nextId = 1;

  constructor(
    public readonly id:    number = BaseProduct._nextId++,
    public           name:  string,
    protected        price: number
  ) {}

  // Abstract — each subclass MUST implement
  abstract calculateShipping(): number;
  abstract get category(): string;

  // Concrete — shared by all
  get totalCost(): number {
    return this.price + this.calculateShipping();
  }

  toString(): string {
    return `[${this.category}] #${this.id} ${this.name} — $${this.totalCost.toFixed(2)}`;
  }
}

// ═══ PHYSICAL PRODUCT ════════════════════════════════════════
class PhysicalProduct extends BaseProduct {
  constructor(
    name: string,
    price: number,
    private weightKg: number
  ) {
    super(undefined, name, price);  // let base assign id
  }

  calculateShipping(): number {
    return this.weightKg * 2.5;  // $2.50 per kg
  }

  get category(): string { return "Physical"; }
}

// ═══ DIGITAL PRODUCT ─────────────────────────────────────────
class DigitalProduct extends BaseProduct {
  constructor(name:string, price:number) {
    super(undefined, name, price);
  }

  calculateShipping(): number { return 0; }  // free!
  get category(): string             { return "Digital"; }
}

// ═══ POLYMORPHIC USAGE ───────────────────────────────────────
const cart: BaseProduct[] = [
  new PhysicalProduct("Laptop",   999, 2.1),
  new DigitalProduct("E-Book",   12.99),
  new PhysicalProduct("Keyboard", 149, 0.8),
];

const total = cart.reduce((sum, p) => sum + p.totalCost, 0);
cart.forEach(p => console.log(p.toString()));
// [Physical] #1 Laptop — $1004.25
// [Digital]  #2 E-Book — $12.99
// [Physical] #3 Keyboard — $151.00
💡
super() Must Be First in Constructors

In TypeScript (and JS), you cannot access this in a subclass constructor before calling super(). The parent must initialize first. The TypeScript compiler will catch this as an error: "super must be called before accessing this."

POLYMORPHISM — ONE NAME, MANY FORMS

polymorphism.ts — method overriding + type narrowing
// Shape hierarchy with runtime polymorphism
abstract class Shape {
  constructor(public readonly color: string = "black") {}
  abstract get area():      number;
  abstract get perimeter(): number;
  abstract scale(factor: number): Shape;  // returns same type

  describe(): string {
    return `${this.color} ${this.constructor.name}:
      area=${this.area.toFixed(2)},
      perim=${this.perimeter.toFixed(2)}`;
  }
}

class Circle extends Shape {
  constructor(public radius: number, color?: string) {
    super(color);
  }
  get area():      number { return Math.PI * this.radius ** 2; }
  get perimeter(): number { return 2 * Math.PI * this.radius; }
  scale(f:number):  Circle  { return new Circle(this.radius * f, this.color); }
}

class Rectangle extends Shape {
  constructor(
    public width: number,
    public height: number,
    color?: string
  ) { super(color); }

  get area():      number    { return this.width * this.height; }
  get perimeter(): number    { return 2 * (this.width + this.height); }
  scale(f:number):  Rectangle {
    return new Rectangle(this.width*f, this.height*f, this.color); }
}

// ── Type narrowing with instanceof ───────────────────────────
function getDiameter(s: Shape): string {
  if (s instanceof Circle) {
    return `Diameter: ${(s.radius * 2).toFixed(2)}`;  // s is Circle here!
  }
  return "Not a circle";
}

// ── Polymorphic array — one loop, different behaviors ────────
const shapes: Shape[] = [
  new Circle(5, "red"),
  new Rectangle(4, 6, "blue"),
  new Circle(3, "green"),
];

const totalArea = shapes
  .reduce((sum, s) => sum + s.area, 0);  // dynamic dispatch!
console.log(`Total area: ${totalArea.toFixed(2)}`);

ABSTRACTION — HIDE THE COMPLEXITY

abstraction.ts — abstract classes + template method pattern
// The Template Method Pattern — abstraction in action
abstract class DataProcessor {

  // Template method — defines the ALGORITHM skeleton
  // Subclasses override individual steps, not the whole thing
  final process(data: string): string {
    const raw       = this.readData(data);
    const validated = this.validate(raw);
    const processed = this.transform(validated);
    return this.format(processed);
  }

  // Hooks — subclasses override these
  protected abstract readData(input:   string): string;
  protected abstract transform(data:    string): string;

  // Default implementations — can be overridden
  protected validate(data: string): string {
    if (!data.trim()) throw new Error("Empty data");
    return data;
  }

  protected format(data: string): string {
    return data.trim();
  }
}

// Concrete CSV Processor
class CsvProcessor extends DataProcessor {
  protected readData(input: string): string {
    return input.replace(/\r\n/g, "\n");  // normalize newlines
  }

  protected transform(data: string): string {
    const rows = data.split("\n");
    const headers = rows[0].split(",");
    return `CSV with ${rows.length} rows, ${headers.length} cols`;
  }

  protected format(data: string): string {
    return `📊 RESULT: ${super.format(data)}`;  // calls parent!
  }
}

class JsonProcessor extends DataProcessor {
  protected readData(input: string): string {
    JSON.parse(input);  // throws if invalid JSON
    return input;
  }

  protected transform(data: string): string {
    const parsed = JSON.parse(data);
    return JSON.stringify(parsed, null, 2);  // pretty print
  }
}

const csv  = new CsvProcessor();
const json = new JsonProcessor();
console.log(csv.process("name,age\nAlice,30"));

INTERFACES — TYPESCRIPT'S SUPERPOWER

TypeScript interfaces are pure type contracts — they exist only at compile time and have zero runtime cost. They enable structural typing (duck typing with validation).

interfaces-complete.ts
// ── Basic interface ───────────────────────────────────────────
interface Serializable {
  serialize():                  string;
  deserialize(data: string):  void;
}

// ── Interface with optional + readonly props ─────────────────
interface UserProfile {
  readonly id:      number;
  username:          string;
  email:             string;
  bio?:               string;          // optional
  avatar?:            string;
  createdAt:         Date;
  toDisplayName():  string;
}

// ── Interface extending multiple interfaces ───────────────────
interface AdminProfile extends UserProfile, Serializable {
  permissions:  string[];
  lastLogin:    Date;
  revokeUser(userId: number): void;
}

// ── Class implementing multiple interfaces ────────────────────
class AdminUser implements AdminProfile {
  readonly id:     number;
  username:         string;
  email:            string;
  bio?:              string;
  avatar?:           string;
  createdAt:        Date;
  permissions:  string[];
  lastLogin:    Date;

  constructor(id:number, username:string, email:string) {
    this.id          = id;
    this.username    = username;
    this.email       = email;
    this.createdAt   = new Date();
    this.lastLogin   = new Date();
    this.permissions = ["read", "write", "admin"];
  }

  toDisplayName(): string     { return `@${this.username} (Admin)`; }
  serialize():     string     { return JSON.stringify(this); }
  deserialize(d:string): void {
    const parsed = JSON.parse(d);
    Object.assign(this, parsed);
  }
  revokeUser(uid:number): void {
    console.log(`Admin ${this.username} revoked user #${uid}`);
  }
}

// ── Structural typing (duck typing) ─────────────────────────
function printUser(user: UserProfile): void {
  console.log(user.toDisplayName());
}

// Works for AdminUser (satisfies UserProfile structurally)
printUser(new AdminUser(1, "alice", "a@b.com"));

GENERICS — TYPE-SAFE REUSABILITY

Generics let you write classes and functions that work with any type while maintaining full type safety. No more casting, no more any.

generics-oop.ts — generic classes and constraints
// ── Generic Stack ─────────────────────────────────────────────
class Stack<T> {
  private items: T[] = [];

  push(item: T): void      { this.items.push(item); }
  pop(): T | undefined    { return this.items.pop(); }
  peek(): T | undefined   { return this.items.at(-1); }
  get size(): number       { return this.items.length; }
  get isEmpty(): boolean   { return this.items.length === 0; }

  // toArray returns a typed copy — no 'any'!
  toArray(): readonly T[] { return [...this.items]; }
}

// ── Generic Repository (DB pattern) ──────────────────────────
interface Entity {
  id: number;
}

class Repository<T extends Entity> {  // T must have an id
  private store = new Map<number, T>();

  save(entity: T): T {
    this.store.set(entity.id, entity);
    return entity;
  }

  findById(id: number): T | undefined {
    return this.store.get(id);
  }

  findAll(): T[] {
    return Array.from(this.store.values());
  }

  delete(id: number): boolean {
    return this.store.delete(id);
  }

  // Generic method with constraint — find with any property
  findWhere<K extends keyof T>(key: K, value: T[K]): T[] {
    return this.findAll().filter(e => e[key] === value);
  }
}

// ── Usage — fully typed, no 'any' anywhere ───────────────────
interface Product extends Entity {
  name:  string;
  price: number;
}

const repo = new Repository<Product>();
repo.save({ id: 1, name: "Widget", price: 9.99 });
repo.save({ id: 2, name: "Gadget", price: 19.99 });

const cheap = repo.findWhere("price", 9.99);  // TypeScript knows this returns Product[]
const nums  = new Stack<number>();
nums.push(1);
// nums.push("hello")  ← TS ERROR: Argument of type 'string' not assignable to 'number'

Mixins — TypeScript's Multiple Inheritance Solution

mixins.ts
// A mixin is a function that takes a class and returns an extended class
type Constructor<T = {}> = new (...args: any[]) => T;

// Mixin 1: Timestamped
function Timestamped<TBase extends Constructor>(Base: TBase) {
  return class extends Base {
    createdAt = new Date();
    updatedAt = new Date();

    touch(): void { this.updatedAt = new Date(); }
  };
}

// Mixin 2: Activatable
function Activatable<TBase extends Constructor>(Base: TBase) {
  return class extends Base {
    isActive = true;
    activate():   void { this.isActive = true; }
    deactivate(): void { this.isActive = false; }
  };
}

// Base class
class User {
  constructor(public name: string) {}
}

// Apply BOTH mixins — effectively "multiple inheritance"!
class TimestampedActivatableUser extends
  Timestamped(Activatable(User)) {}

const user = new TimestampedActivatableUser("Alice");
console.log(user.name);       // Alice (from User)
console.log(user.isActive);   // true (from Activatable)
console.log(user.createdAt);  // Date (from Timestamped)
user.touch();
user.deactivate();

Decorators — Metadata-Driven OOP

decorators.ts — experimentalDecorators required
// Method decorator — logs every call
function Log(
  target: object,
  key: string,
  descriptor: PropertyDescriptor
): PropertyDescriptor {
  const original = descriptor.value;

  descriptor.value = function(...args: unknown[]) {
    console.log(`📞 ${key}(${args.join(", ")})`);
    const result = original.apply(this, args);
    console.log(`✅ ${key} returned:`, result);
    return result;
  };
  return descriptor;
}

// Property decorator — marks as required
function Required(target:object, key:string) {
  let val: unknown;
  Object.defineProperty(target, key, {
    get() { return val; },
    set(v: unknown) {
      if (v === null || v === undefined || v === "")
        throw new Error(`${key} is required`);
      val = v;
    }
  });
}

class OrderService {
  @Required
  apiKey!: string;

  @Log
  createOrder(userId:number, amount:number): string {
    return `order-${userId}-${amount}`;
  }
}

const svc = new OrderService();
svc.apiKey = "secret-123";
svc.createOrder(42, 99.99);
// 📞 createOrder(42, 99.99)
// ✅ createOrder returned: order-42-99.99

SOLID Principles in TypeScript

🧱

S — Single Responsibility

A class does ONE thing. Split UserService into UserAuthService + UserProfileService.

🔓

O — Open/Closed

Open for extension, closed for modification. Add new shapes via inheritance, don't edit the base Shape class.

🔄

L — Liskov Substitution

A subclass should be usable wherever the parent is expected. Don't throw NotImplemented in overrides.

🎯

I — Interface Segregation

Many small interfaces over one big one. A Readable and Writable instead of one ReadWrite.

🔌

D — Dependency Inversion

Depend on abstractions (interfaces), not concretions. Pass ILogger not ConsoleLogger to constructors.

Library Management System
Easy
Create a Book class with properties: isbn (readonly), title, author, isAvailable. Create a Library class with methods: addBook(), checkOut(isbn), returnBook(isbn), searchByAuthor(name). Use encapsulation, getters, and proper TypeScript types throughout.
View Solution
library.ts
class Book {
  private _isAvailable: boolean = true;
  private _borrowedBy?: string;

  constructor(
    public readonly isbn:   string,
    public           title:  string,
    public readonly author: string
  ) {}

  get isAvailable(): boolean         { return this._isAvailable; }
  get borrowedBy():  string | undefined { return this._borrowedBy; }

  checkOut(borrower: string): boolean {
    if (!this._isAvailable) return false;
    this._isAvailable = false;
    this._borrowedBy  = borrower;
    return true;
  }

  returnBook(): void {
    this._isAvailable = true;
    this._borrowedBy  = undefined;
  }

  toString(): string {
    const status = this._isAvailable
      ? "Available"
      : `Borrowed by ${this._borrowedBy}`;
    return `"${this.title}" by ${this.author} [${status}]`;
  }
}

class Library {
  private books = new Map<string, Book>();

  addBook(book: Book): void {
    if (this.books.has(book.isbn))
      throw new Error(`ISBN ${book.isbn} already exists`);
    this.books.set(book.isbn, book);
  }

  checkOut(isbn:string, borrower:string): boolean {
    return this.books.get(isbn)?.checkOut(borrower) ?? false;
  }

  returnBook(isbn: string): void {
    this.books.get(isbn)?.returnBook();
  }

  searchByAuthor(name: string): Book[] {
    return [...this.books.values()]
      .filter(b => b.author.toLowerCase().includes(name.toLowerCase()));
  }

  get totalBooks():     number { return this.books.size; }
  get availableBooks(): number {
    return [...this.books.values()].filter(b => b.isAvailable).length;
  }
}

Generic Event Emitter
Medium
Build a type-safe EventEmitter<T> class where T is a map of event name → handler type. It should have on(event, handler), off(event, handler), emit(event, ...args), and once(event, handler). Fully typed — no any allowed. The compiler should catch calling emit with wrong arguments.
View Solution
event-emitter.ts
type EventMap = Record<string, (...args: any[]) => void>;

class TypedEventEmitter<T extends EventMap> {
  private listeners = new Map<keyof T, Set<Function>>();

  on<K extends keyof T>(event: K, handler: T[K]): this {
    if (!this.listeners.has(event))
      this.listeners.set(event, new Set());
    this.listeners.get(event)!.add(handler);
    return this;
  }

  off<K extends keyof T>(event: K, handler: T[K]): this {
    this.listeners.get(event)?.delete(handler);
    return this;
  }

  emit<K extends keyof T>(event: K, ...args: Parameters<T[K]>): void {
    this.listeners.get(event)?.forEach(h => h(...args));
  }

  once<K extends keyof T>(event: K, handler: T[K]): this {
    const wrapper = (...args: Parameters<T[K]>) => {
      handler(...args);
      this.off(event, wrapper as T[K]);
    };
    return this.on(event, wrapper as T[K]);
  }
}

// Usage — fully typed event map
type AppEvents = {
  login:   (userId: number, email: string) => void;
  logout:  (userId: number) => void;
  message: (text: string, from: string) => void;

// ── Strategy interface ───────────────────────────────────────
interface PaymentStrategy {
  readonly name:   string;
  validate():     boolean;
  pay(amount: number): Promise<PaymentResult>;
}

interface PaymentResult {
  success:   boolean;
  transId:  string;
  message:  string;
}

// ── Concrete strategies ──────────────────────────────────────
class CreditCardStrategy implements PaymentStrategy {
  readonly name = "Credit Card";

  constructor(
    private cardNumber: string,
    private expiry:     string,
    private cvv:        string
  ) {}

  validate(): boolean {
    return /^\d{16}$/.test(this.cardNumber)
        && /^\d{3}$/.test(this.cvv);
  }

  async pay(amount: number): Promise<PaymentResult> {
    if (!this.validate())
      throw new Error("Invalid card details");
    // Simulate API call
    await new Promise(r => setTimeout(r, 100));
    return {
      success:  true,
      transId: `CC-${Date.now()}`,
      message: `Charged $${amount} to card ****${this.cardNumber.slice(-4)}`
    };
  }
}

class PayPalStrategy implements PaymentStrategy {
  readonly name = "PayPal";

  constructor(private email: string) {}

  validate(): boolean {
    return /^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(this.email);
  }

  async pay(amount: number): Promise<PaymentResult> {
    await new Promise(r => setTimeout(r, 200));
    return {
      success:  true,
      transId: `PP-${Date.now()}`,
      message: `$${amount} sent from PayPal (${this.email})`
    };
  }
}

// ── Context class ─────────────────────────────────────────────
class ShoppingCart {
  private items: Array<{ name: string; price: number }> = [];
  private strategy: PaymentStrategy;

  constructor(strategy: PaymentStrategy) {
    this.strategy = strategy;
  }

  // Strategy is swappable at runtime!
  setPaymentStrategy(s: PaymentStrategy): void {
    this.strategy = s;
  }

  addItem(name:string, price:number): void {
    this.items.push({ name, price });
  }

  get total(): number {
    return this.items.reduce((s,i) => s + i.price, 0);
  }

  async checkout(): Promise<PaymentResult> {
    console.log(`Paying $${this.total.toFixed(2)} via ${this.strategy.name}`);
    return this.strategy.pay(this.total);
  }
}

Q01 What is the difference between interface and type in TypeScript OOP?TypeScript +

Interface: Defines the shape of an object and supports declaration merging (multiple declarations of the same interface are merged). Best for defining contracts for classes and object shapes that others will implement or extend. You can re-open an interface across files.

Type alias: More versatile — can represent unions, intersections, tuples, primitives. Cannot be re-opened after declaration. Better for computed types, utility types, or when you need union/intersection types.

interface vs type
// Interface — can be re-opened, implements works perfectly
interface Animal { name: string; }
interface Animal { age: number; }  // merged!
class Dog implements Animal { name="Rex"; age=3; }

// Type — union support, can't re-open
type Result = string | number;  // impossible with interface
type Readonly<T> = { readonly [K in keyof T]: T[K] };

Rule of thumb: Use interface for class contracts and public APIs. Use type for everything else.

Q02 What is the difference between abstract class and interface?Core OOP +

Both define contracts that subclasses must satisfy, but they differ fundamentally:

Abstract class: Can have concrete implemented methods, a constructor, instance fields, and access modifiers. A class can only extend ONE abstract class. Use when subclasses share implementation code.

Interface: Zero runtime cost — erased at compile time. No constructor, no field storage, only method signatures (plus default methods in TS). A class can implement MANY interfaces. Use for defining capabilities.

when to use which
// Abstract: shared code + is-a relationship
abstract class Vehicle {
  protected fuel = 100;   // shared state
  refuel() { this.fuel = 100; }  // shared behavior
  abstract drive(): void;  // must implement
}

// Interface: capability (can-do) contract
interface Flyable {
  fly(): void;
}

// FlyingCar: IS-A Vehicle AND CAN fly
class FlyingCar extends Vehicle implements Flyable {
  drive() { console.log("driving"); }
  fly()   { console.log("flying"); }
}
Q03 Explain TypeScript's structural typing. How does it differ from nominal typing?TypeScript +

Structural typing (TypeScript): Type compatibility is determined by the shape of the object — its properties and methods. If object A has all the members that type B requires, A is compatible with B, even if they have no inheritance relationship.

Nominal typing (Java, C#): Type compatibility is determined by explicit declarations. Two classes with identical structure are NOT compatible unless one extends the other or implements the same interface.

structural typing demo
interface Named { name: string; }

class Cat {
  name = "Whiskers";
  lives = 9;
}

function greet(n: Named): void {
  console.log(n.name);
}

// Cat doesn't implement Named — but it HAS a name property
// So TypeScript accepts it! (structural, not nominal)
greet(new Cat());            // ✓ works!
greet({ name: "raw object" });  // ✓ also works!
Q04 What is method overriding vs method overloading? Are both supported in TypeScript?Core OOP +

Method Overloading — same method name, different parameter types/counts. TypeScript supports it via overload signatures (multiple signatures + one implementation body). This is compile-time (static) polymorphism.

Method Overriding — a subclass provides a new implementation for a parent class method. TypeScript supports it fully with override keyword (TS 4.3+). This is runtime (dynamic) polymorphism.

overload vs override
// OVERLOADING — multiple signatures, one body
class Formatter {
  format(value: string): string;
  format(value: number, decimals: number): string;
  format(value: string | number, decimals?: number): string {
    return typeof value === "number"
      ? value.toFixed(decimals ?? 2)
      : value.trim();
  }
}

// OVERRIDING — subclass redefines parent method
class Animal    { sound(): string { return "..."; } }
class Dog extends Animal {
  override sound(): string { return "Woof"; }  // TS 4.3+ override keyword
}
Q05 What are TypeScript Decorators? Give a real-world use case.Advanced +

Decorators are functions that attach to classes, methods, properties, or parameters and can modify behavior via metadata. They use the @name syntax. They're heavily used in Angular, NestJS, and TypeORM.

Real-world use cases: Route registration (@Get('/users')), database column mapping (@Column()), authentication guards (@UseGuards(AuthGuard)), validation (@IsEmail()), memoization, logging, retry logic.

memoize decorator
function Memoize(
  _target: object,
  key: string,
  descriptor: PropertyDescriptor
): PropertyDescriptor {
  const original = descriptor.value;
  const cache = new Map<string, unknown>();

  descriptor.value = function(...args: unknown[]) {
    const cacheKey = JSON.stringify(args);
    if (cache.has(cacheKey)) return cache.get(cacheKey);
    const result = original.apply(this, args);
    cache.set(cacheKey, result);
    return result;
  };
  return descriptor;
}

class MathUtils {
  @Memoize
  fibonacci(n: number): number {
    if (n <= 1) return n;
    return this.fibonacci(n-1) + this.fibonacci(n-2);
  }
}
